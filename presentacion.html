Paso 1 — Namespace y Seguridad
El namespace monitoring necesita etiquetas especiales para Promtail (requiere root)
# 00-namespace.yaml apiVersion: v1 kind: Namespace metadata: name: monitoring labels: pod-security.kubernetes.io/enforce: privileged pod-security.kubernetes.io/warn: privileged pod-security.kubernetes.io/audit: privileged
🔐 PSA (Pod Security Admission)
Sin estas etiquetas, Promtail y Tekton son bloqueados por el control de seguridad de Kubernetes
# Aplicar y verificar kubectl apply -f monitoring/00-namespace.yaml kubectl get namespace monitoring \ --show-labels # También para Tekton kubectl label namespace tekton-pipelines \ pod-security.kubernetes.io/enforce=privileged \ --overwrite
¿Por qué privileged?
Promtail: necesita leer /var/log del host (runAsUser: 0)

Tekton buildkit: necesita montar el socket de Docker para construir imágenes

Pipelines nativos en Kubernetes — sin Jenkins, sin agentes externos
# Instalar Tekton Pipelines kubectl apply -f https://storage.googleapis.com/\ tekton-releases/pipeline/latest/release.yaml # Instalar Tekton Dashboard (opcional) kubectl apply -f https://storage.googleapis.com/\ tekton-releases/dashboard/latest/release.yaml # Verificar instalación kubectl get pods -n tekton-pipelines # Crear secret Docker Hub kubectl create secret docker-registry \ docker-credentials \ --docker-server=https://index.docker.io/v1/ \ --docker-username=TU_USUARIO \ --docker-password=TU_PAT \ -n tekton-pipelines # Verificar Tasks disponibles kubectl get tasks -n tekton-pipelines
# Lanzar PipelineRun kubectl create -f - <<EOF apiVersion: tekton.dev/v1 kind: PipelineRun metadata: generateName: build-run- namespace: tekton-pipelines spec: pipelineRef: name: build-and-deploy params: - name: image value: USUARIO/app:latest - name: git-url value: https://github.com/USUARIO/repo workspaces: - name: shared-data volumeClaimTemplate: spec: accessModes: [ReadWriteOnce] storageClassName: nfs-csi resources: requests: storage: 500Mi EOF # Monitorear kubectl get pipelineruns -n tekton-pipelines -w

Flujo de Observabilidad Completo
Cómo se conectan todos los componentes en un incidente real
1
Detección — Prometheus
Scraping cada 15s detecta que kube_deployment_status_replicas_available < kube_deployment_spec_replicas. La alerta pasa a FIRING.
2
Notificación — AlertManager
AlertManager agrupa la alerta, espera group_wait: 5s y envía al receiver configurado (Slack, webhook, email, PagerDuty).
3
Investigación — Loki
Grafana → Explore → Loki: {namespace="demo"} |= "error" para ver qué ocurrió en los logs del pod afectado.
4
Root Cause — Tempo
Si el pod tiene instrumentación OTLP, ver las trazas en Tempo para identificar qué operación falló y cuánto tardó.
5
Resolución — ArgoCD
Fix en el código → git push → ArgoCD auto-sync despliega la corrección. Prometheus detecta que la alerta se resuelve y AlertManager envía "RESOLVED".

Troubleshooting — Errores Comunes
Problemas que encontramos y cómo resolverlos
🔴 Grafana CrashLoopBackOff — subPath
Error: not a directory al montar dashboards

Causa: Montar con subPath dentro de un directorio ya montado

Fix: Poner TODOS los dashboards en UN SOLO ConfigMap, montarlo sin subPath
🔴 Tempo CrashLoopBackOff — /tempo
Error: /tempo: is a directory

Causa: /tempo en la imagen ES el binario ejecutable

Fix: Usar mountPath: /var/tempo y emptyDir (no PVC NFS)
🔴 Promtail — "no files to tail"
Causa: kubernetes_sd con NODE_NAME regex no filtra bien

Fix: Usar static_configs con glob directo:
/var/log/pods/*/*/*.log
🔴 Tekton — PodSecurity violation
Error: pods "..." is forbidden

Fix:
kubectl label namespace \ tekton-pipelines \ pod-security.kubernetes.io/\ enforce=privileged --overwrite
🔴 Prometheus NFS lockfile
Error: lock DB directory: resource temporarily unavailable

Fix: Agregar flag al deployment:
--storage.tsdb.no-lockfile
🔴 GitHub bloqueó push — secrets
Error: Push rejected — secret scanning

Fix: Remover credenciales del archivo, crear nuevo PAT, limpiar historial git. Nunca commitear tokens/passwords

Queries de Diagnóstico — Explore
Pegar en Grafana → Explore → seleccionar datasource → Code → ejecutar
¿Qué pod consume más CPU?
topk(10, sum by(namespace, pod) ( rate(container_cpu_usage_seconds_total{ container!="",container!="POD"}[2m]) ) )
¿Qué pod consume más RAM?
topk(10, sum by(namespace, pod) ( container_memory_working_set_bytes{ container!="",container!="POD"} ) )
¿Pods en CrashLoopBackOff?
increase( kube_pod_container_status_restarts_total[15m] ) > 3
¿Deployments sin réplicas?
kube_deployment_status_replicas_available < kube_deployment_spec_replicas
¿Disco libre por nodo?
100 - ( node_filesystem_avail_bytes{ mountpoint="/",fstype!="tmpfs"} / node_filesystem_size_bytes{ mountpoint="/",fstype!="tmpfs"} * 100 )
¿Nodo con menos RAM libre?
bottomk(4, node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes * 100 )
Logs de errores — namespace demo [Loki]
{namespace="demo"} |~ "(?i)(error|exception|fatal)"
Tasa de errores por namespace [Loki]
sum by(namespace) ( rate({namespace=~".+"} |~ "(?i)(error|exception)" [1m]) )
