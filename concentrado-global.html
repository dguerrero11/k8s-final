Fundamentos de Kubernetes
Arquitectura, objetos esenciales y kubectl básico
Arquitectura del Cluster
🧠 Control Plane (master)
🔌
kube-apiserver
Punto de entrada a todo el cluster. Todo pasa por aquí.
🗄️
etcd
Base de datos clave-valor. Guarda el estado completo del cluster.
📅
kube-scheduler
Decide en qué nodo corre cada Pod (CPU, RAM, afinidad).
🔄
controller-manager
Mantiene el estado deseado (ej: si un Pod muere, lo recrea).
🖥️ Worker Nodes
🤖
kubelet
Agente en cada nodo. Recibe órdenes del API server y gestiona los pods.
🌐
kube-proxy
Gestiona las reglas de red (iptables/ipvs) para que los Services funcionen.
🐳
container runtime
Ejecuta los contenedores. En nuestro cluster: containerd.
🔗
CNI (Flannel/Calico)
Plugin de red. Asigna IPs a los Pods y permite comunicación entre nodos.
Objetos Esenciales
Objeto¿Qué hace?Cuándo usarlo
PodUnidad mínima. Uno o más contenedores con red y almacenamiento compartidoRaramente directo — usa Deployment
DeploymentGestiona réplicas de Pods. Soporta rolling updates y rollbackApps stateless (API, frontend)
StatefulSetPods con identidad estable y almacenamiento persistente por podBases de datos (PostgreSQL, Redis)
DaemonSetUn Pod por nodo del clusterAgentes (Promtail, Node Exporter)
ServiceExpone Pods con IP estable y balanceo de cargaSiempre, junto a Deployment
ConfigMapConfiguración no sensible como archivos o variablesConfig de apps, archivos YAML
SecretDatos sensibles codificados en base64Passwords, tokens, certificados
NamespaceAislamiento lógico dentro del clusterSeparar entornos o equipos
Job / CronJobTarea única o programadaMigraciones, backups, reportes
Tipos de Service
ClusterIP
IP interna solo accesible dentro del cluster.

Uso: comunicación entre microservicios. Ej: postgres.tasks.svc:5432
NodePort
Expone el servicio en un puerto fijo de cada nodo (30000–32767).

Uso: acceso desde fuera del cluster en laboratorio. Ej: :30900
LoadBalancer
Crea un load balancer externo (cloud provider).

Uso: producción en AWS/GCP/Azure. Asigna IP pública automáticamente.

# Ver recursos kubectl get pods,deployments,services,pvc -n <namespace> kubectl get nodes -o wide kubectl describe pod <nombre> -n <namespace> kubectl logs <pod> -n <namespace> --tail=50 -f # Crear / aplicar kubectl apply -f archivo.yaml kubectl apply -f directorio/ # Escalar kubectl scale deployment mi-app -n tasks --replicas=3 # Ejecutar comandos en un pod kubectl exec -it <pod> -n <ns> -- bash # Port forward kubectl port-forward svc/grafana 3000:3000 -n monitoring # Ver eventos del namespace (diagnóstico) kubectl get events -n <namespace> --sort-by='.lastTimestamp' # Eliminar kubectl delete pod <nombre> -n <namespace> kubectl delete -f archivo.yaml

Almacenamiento — PV, PVC y CSI Driver
Persistencia de datos en Kubernetes con NFS
Conceptos de almacenamiento
💾
PersistentVolume (PV)
Recurso de almacenamiento en el cluster. Lo crea el admin o el CSI Driver automáticamente.
📋
PersistentVolumeClaim (PVC)
Solicitud de almacenamiento hecha por una app. "Quiero 5Gi con acceso lectura-escritura".
🏷️
StorageClass
Define el "tipo" de almacenamiento y cómo provisionarlo automáticamente (dynamic provisioning).
🔌
CSI Driver
Container Storage Interface. Plugin que conecta K8s con el storage real (NFS, AWS EBS, GCP PD).
# StorageClass NFS (ya instalada) apiVersion: storage.k8s.io/v1 kind: StorageClass metadata: name: nfs-csi provisioner: nfs.csi.k8s.io parameters: server: 192.168.109.210 share: /srv/nfs/k8s reclaimPolicy: Retain --- # PVC — la app pide 5Gi apiVersion: v1 kind: PersistentVolumeClaim metadata: name: mi-datos namespace: tasks spec: accessModes: [ReadWriteOnce] storageClassName: nfs-csi resources: requests: storage: 5Gi
Access Modes
ModeAbreviaciónSignificadoUso típico
ReadWriteOnceRWOMontado en lectura-escritura por UN nodoPostgreSQL, MySQL
ReadOnlyManyROXMontado en solo lectura por MUCHOS nodosAssets estáticos
ReadWriteManyRWXMontado en lectura-escritura por MUCHOS nodosNFS compartido, logs

Tipos de métricas Prometheus
📈
Counter
Solo sube. Ej: tasks_created_total. Para saber cuántas veces pasó algo. Usa rate() para ver la velocidad.
🌡️
Gauge
Sube y baja. Ej: node_memory_MemFree_bytes. Valor actual en un momento dado.
📊
Histogram
Mide distribución de valores. Ej: latencia de requests. Permite calcular percentiles p50, p95, p99.
📋
Summary
Similar a Histogram pero los percentiles se calculan en el cliente, no en Prometheus.
# PromQL — Queries esenciales # CPU de los nodos (últimos 5 min) 100 - (avg by(instance) ( rate(node_cpu_seconds_total {mode="idle"}[5m])) * 100) # Memoria usada por namespace sum by(namespace) ( container_memory_usage_bytes {container!=""}) # Tasa de errores 5xx de la app rate(flask_http_request_total {status=~"5.."}[5m]) # Replicas disponibles kube_deployment_status_replicas_available {namespace="tasks"}

Tempo + OpenTelemetry — Trazas Distribuidas
El tercer pilar: seguir una request de principio a fin
Conceptos de trazas
🔍 Trace
Registro completo de una request de extremo a extremo. Tiene un trace_id único. Contiene múltiples spans.
📏 Span
Una operación individual dentro de la traza. Tiene inicio, duración y atributos. Los spans se anidan formando un árbol.
🏷️ Atributos
Metadatos del span. Ej: db.system=postgresql, http.method=POST, task.id=42
Instrumentación en Python
# Configuración OpenTelemetry (una sola vez) from opentelemetry import trace from opentelemetry.sdk.trace import TracerProvider from opentelemetry.sdk.trace.export import BatchSpanProcessor from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter resource = Resource(attributes={"service.name": "tasks-api"}) provider = TracerProvider(resource=resource) provider.add_span_processor( BatchSpanProcessor( OTLPSpanExporter(endpoint="http://tempo.monitoring.svc:4318/v1/traces") ) ) trace.set_tracer_provider(provider) tracer = trace.get_tracer(__name__) # Crear spans manuales (anidados) @app.route("/tasks", methods=["POST"]) def create_task(): with tracer.start_as_current_span("create-task") as span: span.set_attribute("task.title", title) with tracer.start_as_current_span("db.insert-task") as db_span: db_span.set_attribute("db.system", "postgresql") db_span.set_attribute("db.operation", "INSERT") # ... ejecutar INSERT ...
TraceQL — Buscar trazas en Tempo
# En Grafana → Explore → Tempo → TraceQL # Todas las trazas del servicio {resource.service.name="tasks-api"} # Trazas de un endpoint específico {resource.service.name="tasks-api" && name="create-task"} # Trazas con error {resource.service.name="tasks-api" && status=error} # Trazas lentas (más de 500ms) {resource.service.name="tasks-api"} | duration > 500ms

AlertManager — Alertas y Notificaciones
Definir condiciones críticas y notificar al equipo correcto
Ciclo de vida de una alerta
⚪ INACTIVE
condición falsa
→
🟡 PENDING
condición true, esperando for:
→
🔴 FIRING
confirmada, AlertManager actúa
→
✅ RESOLVED
condición vuelve a false
Anatomía de una regla
groups: - name: tasks-app-alerts rules: - alert: TasksApiDown # La condición PromQL que evalúa cada 15s expr: kube_deployment_status_replicas_available {namespace="tasks",deployment="tasks-api"} == 0 # Debe cumplirse durante 1 minuto para pasar a FIRING for: 1m labels: severity: critical # se usa para enrutar en AlertManager team: platform annotations: summary: "Task Manager sin réplicas" description: "tasks-api en namespace tasks tiene 0 réplicas"
Las 4 reglas del Task Manager
AlertaSeveridadfor:Qué detecta
TasksApiDowncritical1mtasks-api con 0 réplicas disponibles
TasksPostgresDowncritical1mPostgreSQL StatefulSet sin réplicas listas
TasksApiHighErrorRatewarning2mTasa de errores 5xx > 5% por 2 minutos
TasksPostgresDiskHighwarning5mPVC de PostgreSQL usando > 80% capacidad
Demo completa — Disparar y resolver alertas
# Paso 1: Suspender ArgoCD (para que no restaure automáticamente) kubectl patch application tasks-api -n argocd \ --type=merge -p '{"spec":{"syncPolicy":{"automated":null}}}' # Paso 2: Romper la app (escalar a 0) kubectl scale deployment tasks-api -n tasks --replicas=0 # Paso 3: Esperar y ver PENDING → FIRING (~90s) # http://192.168.109.142:30900/alerts # Paso 4: Romper PostgreSQL kubectl scale statefulset postgres -n tasks --replicas=0 # Paso 5: Ver ambas alertas en AlertManager # http://192.168.109.142:30903 # Paso 6: Restaurar TODO kubectl scale deployment tasks-api -n tasks --replicas=2 kubectl scale statefulset postgres -n tasks --replicas=1 kubectl patch application tasks-api -n argocd \ --type=merge -p '{"spec":{"syncPolicy":{"automated":{"prune":true,"selfHeal":true}}}}'

Tekton CI/CD — Pipelines en Kubernetes
Build, test y push de imágenes directamente en el cluster
Objetos de Tekton
🧩
Task
Unidad de trabajo reutilizable. Serie de Steps (contenedores). Ej: git-clone, kaniko-build.
🔗
Pipeline
Secuencia de Tasks conectadas. Define el orden y los parámetros de cada Task.
▶️
PipelineRun
Ejecución concreta de un Pipeline. Crea los Pods necesarios y los elimina al terminar.
💼
Workspace
Volumen compartido entre Tasks. El código clonado en git-clone lo usa la Task de build.
# Flujo del pipeline tasks-api Task 1: kaniko-build-push - Clona el repo desde Git - Construye la imagen con Kaniko - Push a Docker Hub Task 2: update-image-tag - Actualiza el tag en k8s/02-app.yaml - Git commit + push # ArgoCD detecta el cambio en Git # y despliega la nueva versión automáticamente # → CI/CD completo sin Jenkins!
PipelineRun — Ejecutar el pipeline
# Crear un PipelineRun manual apiVersion: tekton.dev/v1 kind: PipelineRun metadata: name: tasks-api-run-001 namespace: tekton-pipelines spec: pipelineRef: name: tasks-api-pipeline params: - name: image-tag value: "v1.0.3" workspaces: - name: source volumeClaimTemplate: spec: accessModes: [ReadWriteOnce] resources: { requests: { storage: 1Gi } } podTemplate: securityContext: { fsGroup: 65532 } # Aplicar y monitorear kubectl apply -f pipelinerun.yaml -n tekton-pipelines kubectl get pipelinerun -n tekton-pipelines -w

ArgoCD — GitOps
Git como fuente de verdad — el cluster siempre refleja el repositorio
El principio GitOps
📖 Sin GitOps
kubectl apply manual → nadie sabe qué está en el cluster
Cambio de emergencia directo → inconsistencia con Git
Rollback = recordar qué había antes
🔄 Con GitOps (ArgoCD)
Git es la única fuente de verdad
ArgoCD sincroniza automáticamente cada cambio
Rollback = git revert
Application manifest
apiVersion: argoproj.io/v1alpha1 kind: Application metadata: name: tasks-api namespace: argocd spec: project: default source: repoURL: https://github.com/dguerrero11/k8s-final targetRevision: main path: app-tasks/k8s # carpeta con los YAMLs destination: server: https://kubernetes.default.svc namespace: tasks syncPolicy: automated: prune: true # elimina recursos que ya no están en Git selfHeal: true # restaura si alguien cambia el cluster a mano syncOptions: - CreateNamespace=true
Estados de sincronización
Sync StatusHealth StatusSignificado
SyncedHealthy✅ Todo bien. Cluster == Git
OutOfSyncHealthy⚠️ Alguien cambió el cluster manualmente. ArgoCD lo corregirá.
SyncedDegraded🔴 Los recursos están desplegados pero los pods tienen errores
OutOfSyncMissing🔴 Recursos en Git pero no en el cluster

A-3 HPA — Horizontal Pod Autoscaler Avanzado

4

5
Para la carga y observa el scale-down (puede tardar 5 min por estabilización)
A-4 Helm Chart para el Task Manager Avanzado
1
Instala Helm y ejecuta: helm create tasks-chart
2
Convierte los YAMLs de app-tasks/k8s/ en templates con variables ({{ .Values.image.tag }})
3
Define en values.yaml: replicas, image tag, DB credentials, nodePort
4
Despliega: helm install tasks-api ./tasks-chart -n tasks
5
Actualiza solo el tag: helm upgrade tasks-api ./tasks-chart --set image.tag=v1.0.5
